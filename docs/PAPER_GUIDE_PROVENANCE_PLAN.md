# 文献阅读指导原文依据定位重构计划

## 1. 背景

当前“阅读指导”模式的核心诉求已经比较明确：

1. 用户在阅读指导会话里看到的回答，不应是黑盒输出。
2. 回答里凡是直接依据当前文献原文的内容，都应该能一键打开该文献对应的渲染页面，并高亮到具体句子、段落、公式或表格。
3. 如果某段回答只是模型综合、归纳或推断，没有明确原文依据，就不应强行给定位入口。
4. 同一段回答允许映射到原文多处，但交互入口应尽量克制，避免页面被大量按钮污染。

当前实现没有稳定满足这几个目标。问题不是样式细节，也不是再补一点模糊匹配规则，而是定位链路的实现方向不对。

## 2. 当前问题与根因

### 2.1 现状

现在的定位功能基本是“回答生成后，再拿回答文本去猜它对应原文哪一段”：

1. 前端按段落或列表项拆回答文本。
2. 再把这些文本和 `ref_locs`、guide 文档片段、reader anchor 做相似度匹配。
3. 匹配成功才给“定位”按钮。
4. 打开 reader 后，如果还没有稳定锚点，再做一次二次模糊定位。

这套方案的本质是前端事后推断，而不是后端在生成回答时产出可验证的依据关系。

### 2.2 根因

根因有四个：

1. 回答段落没有稳定的“证据块 ID”
   现在回答只是一段字符串，没有和原文块建立主键级别关系。

2. reader 里的锚点不是稳定块绑定
   目前 anchor 更多是“按 markdown 渲染顺序消费”，而不是“原文第 N 个块的固定锚点”。

3. 前端在做两次猜测
   一次猜“该不该显示按钮”，一次猜“按钮点开后应该落在哪里”，两次都不可靠。

4. 系统无法区分“直接依据”和“模型综合”
   所以会出现两种都不理想的情况：
   - 有依据的内容不给入口
   - 没有明确依据的总结也被硬匹配到某段原文

### 2.3 直接后果

1. 有时全部跳到文首。
2. 有时映射到错误段落。
3. 公式、公式解释、方法流程这类高密度内容最容易失真。
4. 某些消息会直接一个定位按钮都没有。
5. 用户无法相信“这句话到底是不是原文依据”。

## 3. 目标

本计划的目标不是“让按钮尽量多出现”，而是“让出现的按钮可信”。

### 3.1 产品目标

1. 阅读指导模式下，回答中的依据定位必须以当前绑定文献为唯一主文档。
2. 回答按句或按信息密度较高的段进行证据映射。
3. 公式和公式解释必须支持定位到原文公式块或紧邻解释块。
4. 一段回答可映射到原文多处。
5. 没有足够证据时宁可不显示定位入口，也不要伪造映射。

### 3.2 工程目标

1. 用稳定 `block_id` 取代前端基于文本的模糊定位。
2. 用后端显式返回的 provenance 数据取代前端事后猜测。
3. reader 的高亮与跳转只依赖稳定块标识，不再依赖 markdown 渲染顺序。
4. 保持与现有普通会话兼容，不影响当前答复主链路。

## 4. 非目标

以下内容不是本阶段重点：

1. 不追求所有普通聊天历史消息都自动补齐精确定位。
2. 不追求跨多篇文献的复杂联合证据映射。
3. 不在这一阶段引入大规模自动评测。
4. 不在这一阶段重做整套 reader 视觉设计。

## 5. 设计原则

1. 证据优先于样式
   先确保“为什么能跳到这里”是正确的，再谈按钮怎么摆。

2. 后端产出事实，前端只做表达
   是否显示定位入口，应由后端返回的证据结构决定，而不是前端猜。

3. 主文档唯一
   阅读指导模式下，定位默认只指向当前绑定文献，不混入其他参考文献。

4. 缺证据就不展示
   如果一段话没有达到映射置信阈值，不应给按钮。

5. 一对多允许，但入口收敛
   同一段回答只保留一个入口；若对应多处原文，在 reader 内切换或并列高亮。

## 6. 目标架构

目标链路应改为：

1. 文献入库后切分成稳定 `SourceBlock`
2. paper guide 检索和精读都围绕 `SourceBlock`
3. 生成回答时，同时生成 `AnswerSegment -> SourceBlock[]` 的依据关系
4. 前端按 `AnswerSegment` 渲染文本和定位入口
5. reader 按 `block_id` 精确滚动和高亮

这意味着“定位”不再是回答后的附加猜测，而是回答结果的一部分。

## 7. 数据模型

### 7.1 SourceBlock

为主文档建立统一块模型，至少包含：

1. `doc_id`
2. `block_id`
3. `anchor_id`
4. `block_type`
   - `heading`
   - `paragraph`
   - `equation`
   - `list_item`
   - `table`
   - `caption`
   - `code`
5. `heading_path`
6. `order_index`
7. `line_start`
8. `line_end`
9. `text`
10. `normalized_text`
11. `tex`
12. `equation_label`
13. `char_start`
14. `char_end`

### 7.2 AnswerSegment

回答不再只是一整段 markdown，而是带证据信息的 segment 列表：

1. `segment_id`
2. `text`
3. `segment_type`
   - `claim`
   - `summary`
   - `equation_explanation`
   - `comparison`
   - `procedure`
4. `evidence_mode`
   - `direct`
   - `synthesis`
   - `none`
5. `evidence_block_ids`
6. `evidence_confidence`
7. `evidence_notes`

### 7.3 ProvenancePack

每条 assistant 消息应附带一个 provenance 结构，便于前端直接消费：

1. `conversation_id`
2. `message_id`
3. `guide_doc_id`
4. `segments`
5. `block_map`
6. `version`

## 8. 后端改造计划

### P0. 停止继续把前端模糊定位当正式方案

目标：

1. 明确当前前端 heuristic 只作为过渡 fallback。
2. 不再围绕“再补几个匹配规则”持续投入。

产出：

1. 计划层面确认后续主线迁移到 provenance 模型。

### P1. 建立主文档稳定块索引

目标：

1. 对阅读指导绑定文献生成稳定 `SourceBlock` 列表。
2. 为段落、公式、标题、表格等生成固定 `block_id` 和 `anchor_id`。

改造点：

1. markdown 解析与切块逻辑
2. 公式单独建块
3. 原文块序号和 heading path 固化
4. reader doc 接口返回块级结构

完成标志：

1. 同一篇文献重复打开时，块 ID 稳定不变。
2. 公式块能单独寻址。

### P2. 重构 reader 文档接口

目标：

1. 不再只返回“markdown + 松散 anchors”。
2. 返回可直接渲染的块级数据。

建议接口输出：

1. `doc_meta`
2. `blocks[]`
3. `anchor_index`
4. `toc`

完成标志：

1. 前端不再需要按渲染顺序分配 anchor。
2. reader 可直接用 `block_id` 和 `anchor_id` 高亮。

### P3. 阅读指导检索切到 block-level

目标：

1. 阅读指导模式只在当前绑定文献内检索。
2. 检索返回的是原文块，而不是散乱片段。

改造点：

1. guide session 的 source scope 强绑定
2. block 检索与 rerank
3. 公式块优先规则
4. 邻接块扩展策略

完成标志：

1. 提问公式时，候选里优先含公式块及其解释块。
2. 提问方法原理时，候选集中在同一文献相关 section 内。

### P4. 生成回答时同步产出证据映射

目标：

1. 模型输出回答时，同时产出 `segments + evidence_block_ids`。

建议方式：

1. 检索阶段先拿到候选块
2. 组织 prompt 时把块 ID 显式注入上下文
3. 要求模型按 segment 输出结构化结果
4. 后处理校验 block_id 是否都属于当前 guide 文档

完成标志：

1. assistant 消息天然带有依据块，而不是前端再猜。
2. `synthesis` 段落可以没有按钮。

### P5. 细化公式映射策略

目标：

1. 公式与公式解释成为一级场景单独处理。

规则：

1. 公式本身优先映射到 `equation` 块。
2. 对公式中变量解释，允许同时映射：
   - 公式块
   - 紧邻解释段
   - 方法 section 内定义块
3. 允许一对多返回。

完成标志：

1. 询问某个公式含义时，能稳定打开原文公式位置。

## 9. 前端改造计划

### P6. 回答按 segment 渲染

目标：

1. 不再对整段 markdown 事后做 snippet 级猜测。
2. 直接消费后端返回的 `segments`。

改造点：

1. Message 数据结构增加 provenance 字段
2. MessageList 渲染 segment
3. MarkdownRenderer 接收 segment-level locate metadata

完成标志：

1. 是否显示按钮由 `evidence_block_ids` 决定。
2. 无需再做前端文本相似度判断。

### P7. 定位入口交互收敛

目标：

1. 一段回答保留一个定位入口。
2. 不让页面被大量锚点图标污染。

建议交互：

1. 仅对有证据的 segment 显示一个小型“定位依据”入口。
2. 如果有多处证据，入口显示 `xN`。
3. hover 或点击时展示简短来源说明：
   - 第几节
   - 段落/公式
   - 是否多处

完成标志：

1. 用户能明确知道“这段话依据的是哪部分原文”。

### P8. reader 精确滚动与高亮

目标：

1. 点开后直接滚动到准确块。
2. 支持单块高亮与多块切换。

改造点：

1. reader DOM 节点带 `data-block-id`
2. 高亮按 `block_id` 触发
3. 多候选切换器
4. 默认滚到主候选，其余作为备选

完成标志：

1. 不再出现“总是跳到开头”。
2. 多处依据时能切换查看。

## 10. 兼容策略

### 10.1 旧消息兼容

旧的阅读指导消息没有 provenance 数据，不能伪装成精确映射。

处理策略：

1. 旧消息默认不显示精确定位入口。
2. 如保留 fallback，应明确标记为“低置信定位”。
3. 新功能效果以新提问产生的新消息为准。

### 10.2 普通会话兼容

普通会话不强制启用这套强绑定模型。

策略：

1. 仅在 `paper_guide` 模式启用主文档精确依据定位。
2. 普通会话继续保留现有 refs 行为，但不承诺逐段精确落点。

## 11. 与现有计划的整合

### 11.1 与答复优化计划的关系

这项工作应并入答复优化主线，但定位为“高置信依据链路重构”，不是文案层优化。

建议新增一个专项分支：

1. `E1` 主文档块级索引
2. `E2` 回答 provenance 协议
3. `E3` reader 精确高亮
4. `E4` 小样本回归

### 11.2 与文献篮计划的关系

文献篮侧的价值在于“选中文献并进入阅读指导”。

这套 provenance 改造完成后，文献篮与阅读指导之间的关系会更清晰：

1. 文献篮负责选文献
2. 阅读指导负责围绕该文献生成可追溯回答
3. 定位能力只依赖绑定文献，不依赖文献篮本身 UI

因此优先级上，当前应先完成 provenance 主链路，再继续细修文献篮交互细节。

## 12. 优先级排序

按收益与必要性排序，建议如下：

1. P1 主文档稳定块索引
2. P2 reader 文档接口块级化
3. P3 阅读指导 block-level 检索
4. P4 回答 provenance 输出
5. P6 segment 渲染
6. P8 reader 精确高亮
7. P5 公式专项增强
8. P7 定位入口交互收敛
9. 兼容清理与旧逻辑降级

理由：

1. 没有稳定块模型，就没有精确锚点。
2. 没有回答时同步产出的 provenance，前端永远只能猜。
3. 公式支持依赖前两项，不应独立硬补。

## 13. 分阶段实施建议

### 第一阶段：打通主链路

范围：

1. `SourceBlock`
2. reader block API
3. guide 检索切到 block 级
4. 回答输出 `segments + evidence_block_ids`

目标：

1. 至少让新生成的阅读指导消息具备可靠定位基础。

### 第二阶段：前端精确消费

范围：

1. segment 渲染
2. 单入口定位
3. reader 精确高亮
4. 多处依据切换

目标：

1. 用户第一次真正看到“这句话依据原文哪里”。

### 第三阶段：专项增强

范围：

1. 公式/变量解释
2. 表格、图注、算法流程块
3. 低置信 fallback 清理

目标：

1. 提高高密度科研内容的可用性。

## 14. 验证方案

遵循当前约束，不做大规模测试，只做最小必要验证。

### 14.1 基础验证

1. `health`
2. `quality_summary`
3. 前后端最小联调

### 14.2 小样本点测

固定用 SCINeRF 这类已有问题样本做点测，至少覆盖：

1. “这篇文章讲了什么”
2. “某个公式是什么意思”
3. “方法核心贡献是什么”

验收标准：

1. 有原文依据的段落出现定位入口
2. 无原文依据的综合段落不出现入口
3. 公式问题能跳到公式块
4. 多处依据能切换
5. 不再出现全部跳文首

## 15. 风险

1. markdown 切块与 reader 渲染不一致
   需要统一块模型来源，避免前后端各切各的。

2. 模型输出 segment 结构不稳定
   需要后处理校验与必要的降级策略。

3. 历史消息无法自动升级
   需要接受“新消息生效，旧消息有限兼容”这个现实约束。

4. 公式提取质量受 markdown 质量影响
   如果原始 md 的公式结构混乱，需要在入库阶段补强。

## 16. 回滚与降级

如果中途任一步未稳定，不应把 heuristic 再次升级成主方案。

降级策略：

1. 新 provenance 链路未就绪时，阅读指导消息默认不显示精确定位入口。
2. 如保留旧 fallback，明确标低置信，不默认展示。

## 17. Definition of Done

满足以下条件，才算这项功能真正完成：

1. 阅读指导新消息按 segment 携带 provenance。
2. 点击定位可以稳定打开绑定文献的正确位置。
3. 公式与公式解释可定位到公式块或对应解释块。
4. 无直接依据的综合段落不显示定位入口。
5. 一段回答对应多处原文时，只保留一个入口并支持切换。
6. 不依赖前端对回答文本进行模糊猜测。

## 18. 下一步建议

下一步不应继续修前端按钮显隐，而应先开始第一阶段：

1. 定义 `SourceBlock` 与 reader block API
2. 明确 assistant 消息的 provenance 协议

这两步完成后，后面的按钮、跳转和高亮才有坚实基础。
