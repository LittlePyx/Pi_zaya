# Literature Classification And Tagging Plan

## 1. Purpose

This document defines the recommended product and architecture plan for adding
paper classification and tagging to the existing library workflow.

It is designed to fit the current codebase structure:

- persistent library records live in `kb/library_store.py`
- library management UI lives in `web/src/pages/LibraryPage.tsx`
- conversation-local citation collection lives in `web/src/components/chat/CiteShelf.tsx`

The goal is to add long-term paper organization without mixing it with
conversation-local temporary annotations.

## 2. Product Decision

The recommended model is:

- low-risk metadata can be filled automatically
- semantic category and tag suggestions should be generated by the system
- the user must own the final saved result

In short:

- `auto` for objective metadata
- `suggest` for semantic organization
- `user-confirmed` for final persistent classification and tags

This is the safest and most scalable model for a research library.

## 3. Why Not Pure Auto Or Pure Manual

### Pure Auto

Pros:

- fast
- low user effort

Cons:

- semantic tags are often subjective
- wrong classification is very annoying at scale
- later correction becomes expensive

### Pure Manual

Pros:

- highest control
- no silent wrong writes

Cons:

- too much work for a growing library
- users stop maintaining it consistently

### Recommended Hybrid

Pros:

- system reduces user effort
- user remains in control
- wrong suggestions do not pollute the library automatically

This should be the default product direction.

## 4. Core Principles

1. Library-level classification must be persistent.
2. Conversation-local shelf tags must remain separate from library tags.
3. User edits must always override system suggestions.
4. Suggestions may be refreshed; user choices must not be overwritten.
5. Classification should attach to stable paper identity, not file path alone.

## 5. Scope Separation

There should be two different layers.

### A. Library-Level Persistent Organization

This is the long-term paper organization layer.

Fields:

- `category`
- `tags`
- `reading_status`
- `note`

This data should be visible and editable from the library page.

### B. Conversation-Level Temporary Annotation

This is the current CiteShelf-style workflow.

Fields:

- temporary shelf tags
- temporary notes
- session-specific organization

This data is useful inside a conversation, but it should not be treated as the
authoritative library taxonomy.

Recommended relationship:

- keep CiteShelf tags as local annotations
- add an explicit action such as `Save to library tags`
- do not auto-sync shelf tags into the library

## 6. Recommended Data Model

The persistent library model should be keyed by `sha1`, not only by `path`.

Reason:

- files can be renamed
- paths can move
- the paper identity should remain stable

### Recommended Tables

#### `paper_meta`

- `sha1 TEXT PRIMARY KEY`
- `category TEXT`
- `reading_status TEXT`
- `note TEXT`
- `user_updated_at REAL`
- `suggestion_updated_at REAL`

#### `paper_tags`

- `sha1 TEXT NOT NULL`
- `tag_norm TEXT NOT NULL`
- `tag_label TEXT NOT NULL`
- `source TEXT NOT NULL`
- `PRIMARY KEY (sha1, tag_norm, source)`

Where `source` is one of:

- `user`
- `suggested`
- `dismissed`

This is better than a single `tags_json` field because:

- filtering is easier
- batch editing is cleaner
- tag stats are easier to compute
- suggestion and user layers stay separate

### Optional Table

#### `paper_suggestions`

If suggestion auditing becomes important, add a separate table:

- `sha1`
- `suggested_category`
- `suggested_tags_json`
- `reason_json`
- `confidence`
- `generator_version`
- `updated_at`

For v1, this can also be stored inside `paper_meta`.

## 7. Final Saved Fields Vs Suggestion Fields

The system should distinguish between:

### Final user-owned fields

- `user_category`
- `user_tags`
- `reading_status`
- `note`

### Suggestion-only fields

- `suggested_category`
- `suggested_tags`
- `dismissed_suggestions`

Display rule:

- show final user result as the active result
- show suggestion chips separately as recommendations

Update rule:

- new suggestions may be added later
- user-owned values must not be overwritten by new suggestion runs

## 8. What Should Be Automatic

Only objective or low-risk fields should be automatic.

Recommended automatic fields:

- title normalization
- authors normalization
- venue
- year
- DOI
- journal / conference indicators
- source path mapping
- basic paper type when confidence is high:
  - `survey`
  - `benchmark`
  - `dataset`
  - `method`

These fields are either objective or close to objective.

## 9. What Should Be Suggested

Semantic organization should be suggested, not auto-applied by default.

Examples:

- category:
  - `NeRF`
  - `3DGS`
  - `SCI`
  - `Video Reconstruction`
  - `Diffusion`
- tags:
  - `single-image`
  - `pose-free`
  - `real-time`
  - `novel-view-synthesis`
  - `view-consistency`
  - `camera-pose`
  - `related-work`

These are useful, but not objective enough to write blindly.

## 10. What Must Be User Editable

The user must be able to:

1. change the primary category
2. add tags
3. remove tags
4. rename a tag label if needed
5. accept a suggested tag
6. dismiss a suggested tag
7. accept a suggested category
8. ignore a suggested category
9. batch edit multiple papers

User edits must have highest priority.

## 11. Suggested UX

### Library Page

This should be the main management surface.

Recommended additions to `LibraryPage`:

- filter bar:
  - `category`
  - `tag`
  - `reading_status`
  - `only unclassified`
  - `only has suggestions`
- row-level display:
  - one category pill
  - up to three tag pills
  - reading status badge
  - suggestion badge if pending
- row actions:
  - `Edit`
  - `Accept suggestions`
  - `Open details`
- batch actions:
  - assign category
  - add tag
  - remove tag
  - set reading status

### Details Drawer Or Side Panel

Each paper should have an editable details panel with:

- final category
- final tags
- reading status
- user note
- suggested category
- suggested tags
- accept / dismiss controls

### Conversation Integration

In CiteShelf or citation popovers:

- keep temporary tags local
- add `Save to library tags`
- add `Set library category`

But do not turn the chat page into the main taxonomy manager.

## 12. Reading Status

Reading status should be an explicit field, not just a tag.

Recommended values:

- `unread`
- `reading`
- `done`
- `revisit`

Reason:

- status is operational workflow
- tags are semantic descriptors
- mixing them reduces filter quality

## 13. Category Strategy

Category should be single-select in v1.

Why:

- it forces one primary home
- browsing stays simple
- UI remains compact

If needed later, support secondary categories through tags rather than true
multi-category assignment.

Recommended interpretation:

- category = primary bucket
- tags = secondary descriptors

## 14. Tag Strategy

Tags should be multi-select and normalized.

Rules:

1. store a normalized internal key
2. preserve a display label
3. dedupe case-insensitively
4. cap extreme per-paper tag counts in UI
5. support batch add and batch remove

Recommended normalization:

- lowercase
- trim whitespace
- collapse repeated separators
- preserve readable display label separately if desired

## 15. Suggestion Engine Strategy

Suggestion generation should be asynchronous.

It should not block:

- upload
- conversion
- conversation open

Recommended trigger points:

1. after a paper is added to the library
2. after citation metadata becomes available
3. after paper-guide analysis has enough signals
4. on explicit user request to regenerate suggestions

### Suggestion Inputs

Potential inputs:

- title
- abstract-like citation summary
- venue
- keywords from source blocks
- paper-guide signals from provenance and guide summaries
- existing user tags across similar papers

### Suggestion Output

Return:

- `suggested_category`
- `suggested_tags`
- optional reasons and confidence

Example:

- category: `SCI`
- reasons:
  - title contains `snapshot compressive image`
  - multiple source-block signals match `SCI`
- tags:
  - `single-image`
  - `camera-pose`
  - `novel-view-synthesis`

## 16. Suggested Acceptance Rules

Default behavior:

- suggestions are shown but not applied automatically

Optional future setting:

- auto-apply only high-confidence low-risk suggestions

Examples of possible auto-apply candidates:

- `survey`
- `dataset`
- `benchmark`

Examples that should remain recommendation-first:

- `NeRF`
- `SCI`
- `3D reconstruction`
- `pose-free`

## 17. API Plan

Recommended backend additions under library routes:

- `GET /api/library/meta`
  - list papers with category, tags, status, suggestions
- `POST /api/library/meta/update`
  - update one paper
- `POST /api/library/meta/batch_update`
  - batch category/tag/status change
- `POST /api/library/meta/suggestions/regenerate`
  - rerun suggestion generation
- `GET /api/library/meta/tag_stats`
  - tag counts for filters and chips

Request identity should prefer:

- `sha1`

Fallback:

- `path`

but the backend should resolve path to `sha1` before writing.

## 18. Frontend Plan

### `web/src/api/library.ts`

Add models and methods for:

- paper meta list
- update
- batch update
- tag stats
- suggestion regeneration

### `web/src/stores/libraryStore.ts`

Extend state with:

- library meta records
- tag stats
- pending suggestion state
- selected filters

### `web/src/pages/LibraryPage.tsx`

Add:

- filter toolbar
- category and tag columns
- bulk edit controls
- details panel

### CiteShelf Integration

Add explicit actions only:

- `Save current shelf tags to library`
- `Save current note to library note`

No silent sync.

## 19. Migration Plan

### Phase A. Schema

1. extend `LibraryStore`
2. create `paper_meta`
3. create `paper_tags`
4. add accessors and update methods

### Phase B. API

1. add list/update/batch routes
2. add tag stats
3. add suggestion trigger route

### Phase C. Library UI

1. show category and tag pills
2. add filter toolbar
3. add edit panel
4. add batch edit

### Phase D. Suggestion Engine

1. add async suggestion generation
2. store suggestions separately from user fields
3. surface accept / dismiss UX

### Phase E. Conversation Hooks

1. CiteShelf `Save to library tags`
2. citation card shortcut
3. optional reader shortcut

## 20. Non-Goals

This plan does not aim to:

1. replace CiteShelf
2. add a full ontology system
3. auto-build perfect paper taxonomy with no user review
4. use `project_id` as the paper classification model

## 21. Recommended V1

If implementation needs to stay minimal, V1 should include only:

1. persistent `category`
2. persistent `user_tags`
3. persistent `reading_status`
4. persistent `note`
5. optional `suggested_category`
6. optional `suggested_tags`
7. library-page filters and edit UI

This is enough to ship real value without overbuilding.

## 22. Final Recommendation

The recommended framework is:

- `library-level persistent classification`
- `conversation-level temporary shelf tagging`
- `system suggestions`
- `user confirmed final result`

The single most important rule is:

`system may suggest, but user owns the final saved category and tags`

That rule will keep the feature useful, trustworthy, and maintainable.
