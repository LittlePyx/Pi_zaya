## 1. 需求整理（按主题归类）

### A. 检索再读（高优先级）

- 提升参考定位准确性
- 命中参考文献后细读（兼顾效率、精度）
- 参考定位细化到文献的 `section / subsection`
- 显示原文片段（需要讨论是否保留，若保留的话显示最相关一句）
- 给出“这篇文献讲了什么 / 与当前问题相关性是什么 / 应该从哪里读起 / 用户进入该文献阅读可收获到什么与提问相关的内容”的摘要与导航建议，若太长点击展开
- `page` 按钮（页码入口）是否保留？（倾向于删去）

### B. 文献阅读与阅读指引（高优先级）

- 增加文献阅读指引能力（从会话入口进入、从参考定位进入）
- 倾向基于 Markdown 阅读，而不是先做完整 PDF 阅读器
- 会话中若提到相应的文献中的地方 / 定位就在附近提供一个小入口，点击后右边可打开 Markdown（以HTML渲染）HTML DOM+锚点 ，支持跳转 + 高亮
- 在阅读页面（以HTML显示的Markdown）支持“划词提问”（监听 `mouseup`，可直接点击出现的小悬浮框“ASK”把选中文本送入提问流程）
- 阅读时可解释文中具体句子/段落含义（偏“边读边问”）

### C. 文献篮 / 引用 UI（高优先级）

- 核对悬浮框里的 source 信息是否与文献实体准确对应
- 二级文献入库时，引用标题与实际文献/Markdown 标题的映射对齐
- 右侧悬浮框会遮挡对话，需要优化交互形态
- 候选方案：
  - 改为可调节宽度右侧栏（类似左侧边栏）
  - 改为单独页面
  - 保留悬浮框但优化显示/消失逻辑（暂时倾向于这个）

### D. Paper Connected（文献关系网络，分阶段做）

- 一级：用户主动上传文献（当前已有基础）
- 二级：把一级文献的 references 做入库（可基于 Crossref）
- 二级库最好做成独立模块，可展开/关闭查看
- 三级：在二级基础上扩展“引用 + 被引”关系，逐步形成 `paper connected` 网站/图谱
- 可在会话中选择打开（？）

### E. 会话能力增强（高优先级）

- 输出与回答相关的文献图片，并结合图注/正文解释
- source 粒度更细（公理、公式这种都需要能显示source）
- 内容质量控制（减少跑偏、提高稳定性）
- 代码/伪代码渲染美化与高亮
- 会话中支持选字触发悬浮框，继续 ASK（类似 GPT 阅读交互）
- Timeline：鼠标悬停显示当时提问内容
- 用户需求：如果用户问到某一篇文章种的某一个定理（公式）该如何解释/证明，需要有定位到的能力。

### F. PDF -> Markdown 质量与工程化（高优先级）

- 用 1980 年前扫描文献验证转换稳定性（判断是否要预处理）
- 用数学/公式密集文献做专项验证
- 评估“翻译后 Markdown”功能是否需要
- 代码过长，需要整理
- 结合阅读场景，把图像抽取能力接入会话输出  

### G. 用户数据 / 项目管理 / 文献管理增强（中高优先级）

- 用户数据库（用户级数据持久化）
- 历史会话管理
- Project 概念（按项目组织文献与对话）
- 文献 tag / 分类
- 文献管理页面 UI + 交互优化（含进度条、转换状态显示）
- 文献详细数据获取（题录/元数据增强）
- 面向领域调研/知识库 LLM 的工作流能力

---

## 2. 总体推进策略（建议）

先做这三件，再扩功能：

1. 打通“检索再读 + 文献阅读（Markdown）”主链路
2. 再扩展 `paper connected` 和高级会话能力

原因：

- 你的核心价值已经不是“能问答”，而是“能快速、可信地读懂文献”
- `检索再读`、`文献阅读引导`、`source 细粒度` 直接提升用户体验和学术可信度
- `paper connected` 很有潜力，但对数据模型和引用质量依赖大，适合在引用定位稳定后推进

## 附录B：跨领域学术情报（Venue Intelligence）与用户主导学习路径（功能草案）

### 1. 功能定位（新增方向）

目标：不仅面向计算机顶会，还面向用户所属具体领域的顶会、顶刊、综合期刊（如 `Nature` / `Science`）进行阶段性整理，并结合用户兴趣与学习目标提供“可操作的学习建议”。

重点定位：

- 这是“辅助决策与导航”功能，不是替用户做死板计划。
- 系统提供候选路径、理由、证据、阅读入口，由用户自己选择/调整/跳过。
- 结合 LLM 提升总结质量、解释能力、个性化推荐质量，但输出必须尽量证据化、可追溯。

一句话定位：

- 用 LLM 做“跨领域学术情报整理 + 个性化学习辅助”，让用户更快知道“最近有什么突破、哪些值得看、先看什么、为什么”。

### 2. 用户体验原则（主打辅助，不替用户做主）

产品原则（必须坚持）：

- 用户主导：系统生成“候选学习路径”，用户可手动删改、替换、重排、标记优先级。
- 证据优先：每个“突破候选 / 推荐论文”都要附理由与依据（摘要句、任务/方法标签、对比信息）。
- 分层输出：先给结论和建议，再允许展开看细节，不让用户一上来被信息淹没。
- 可立即行动：每条推荐都要能一键进入下一步（加入文献篮、加入 Project、开始阅读、问 LLM 解释）。
- 可纠偏：允许用户反馈“这个不相关/太基础/太前沿/换一个”，系统据此更新推荐。

不建议做的事情：

- 自动给用户排死的日程表（例如强制“第1天读A，第2天读B”）
- 不展示依据就下“突破”结论
- 用统一模板硬套所有领域

### 3. 用户如何使用（舒适流程草案）

#### 3.1 初次使用（建立偏好）

用户设置：

- 研究领域（可多选，例如：计算机视觉 / 生物信息 / 材料 / 数学优化 / 医学影像）
- 子方向（自由输入 + 推荐标签）
- 当前目标（补基础 / 跟前沿 / 开题调研 / 写综述 / 复现）
- 阅读偏好（理论优先 / 方法优先 / 实验优先 / 代码优先）
- 时间预算（宽松 / 正常 / 紧凑，或每周小时数）
- 关注来源（顶会 / 顶刊 / 综合刊 / 指定 venue）

说明：

- 这些设置用于“个性化排序与解释”，不是硬性约束，用户可随时改。

#### 3.2 日常使用（学术情报入口）

用户进入 `学术情报 / Venue Intelligence` 页面后：

- 先看到“本期值得关注的方向变化”（按其领域过滤）
- 再看到“突破候选”（附证据与置信度）
- 再看到“适合你先看的论文候选”（不是固定计划）
- 每篇卡片提供：
  - 为什么推荐给你
  - 建议先看哪一部分（如摘要 / 方法 / 实验 / 某 section）
  - 一键加入文献篮 / Project
  - 一键开始阅读
  - 一键让 LLM 解释“它和我当前问题的关系”

#### 3.3 用户主导的学习路径（不是死规定）

建议交互形态：

- “学习路径候选板（Path Board）”而不是“固定课程表”
- 系统给出多个候选路径视角：
  - 入门补底路径
  - 前沿追踪路径
  - 问题导向路径（围绕用户当前项目）
- 用户可以：
  - 拖拽排序
  - 替换某篇论文
  - 标记“已读 / 稍后 / 不感兴趣”
  - 要求 LLM 重生成“更偏理论/更偏工程”的版本

### 4. 跨领域支持的核心抽象（会议/期刊统一）

建议把“顶会/顶刊/综合刊”统一抽象为 `venue`：

- `venue_type`: `conference` / `journal` / `magazine_or_general`
- `venue_name`: 如 `NeurIPS` / `Nature` / `Science`
- `edition`: 如某年某届会议，或期刊某卷某期，或某时间窗口
- `topic_scope`: 用于领域过滤（系统推断 + 用户配置）

这样做的好处：

- 同一套流程可以同时处理“CVPR 2026”与“Nature 最近一个月”
- 后续可以扩到任意领域，不把系统写死在计算机顶会逻辑上

### 5. LLM 在这个功能里的作用（必须显式使用）

这个功能非常适合 LLM，但需要“LLM + 结构化证据”的组合，不建议纯 LLM 自由发挥。

#### 5.1 LLM 可发挥价值的环节

- 题录/摘要标准化辅助（字段清洗、venue 名称归一、主题词提取）
- 主题聚类后的语义命名（给 cluster 起用户可理解的名字）
- “突破候选”说明生成（基于证据生成解释，不是空口总结）
- 个性化推荐理由生成（结合用户目标、基础、历史偏好）
- 阅读入口提示（建议先读哪一节、读完能收获什么）
- 会话联动解释（“这篇和你当前问题的关系是什么”）

#### 5.2 LLM 使用原则（避免幻觉与过度主观）

- 先结构化，再总结：先做检索/聚类/筛选/打分，再让 LLM 解释
- 给 LLM 明确证据输入：摘要、关键词、方法标签、指标变化、用户画像
- 输出标识置信度：`高/中/低` 或分数
- 标注“突破候选”而非“已证实突破”
- 对关键结论保留可追踪来源（标题、摘要句、链接）

### 6. 功能模块草案（产品与工程）

#### 6.1 模块一：学术情报（Digest）

功能介绍：

- 面向指定领域或 venue 的阶段性情报整理（会后总结 / 月度综述）
- 输出热点主题、突破候选、代表论文、趋势变化

实现草案：

- 输入：
  - venue 列表 / 领域过滤条件 / 时间窗口
- 处理：
  - 元数据采集（会议 accepted list / 期刊目录 / Crossref / DBLP / arXiv 等）
  - 标准化入库（允许“无全文仅题录/摘要”）
  - embedding 聚类 + 标签抽取
  - 趋势对比（与往期同 venue 或同领域数据）
  - LLM 生成摘要与解释（基于结构化证据）
- 输出：
  - 主题概览
  - 代表论文卡片
  - 突破候选列表（证据 + 置信度）

#### 6.2 模块二：用户主导学习路径（Path Board）

功能介绍：

- 根据用户领域、目标、基础、时间预算生成“学习路径候选”
- 用户可自由调整，不做强制安排

实现草案：

- 输入：
  - 用户画像（显式设置 + 历史行为）
  - 候选论文池（Digest 结果 + 文献库 + 文献篮 + Project）
- 处理：
  - 前置知识判断（是否需要先看综述/经典）
  - 路径候选生成（入门/前沿/问题导向多版本）
  - LLM 生成每篇推荐理由与建议阅读入口
- 输出：
  - 可编辑路径卡片列表（支持拖拽、替换、标记）
  - 每项的“推荐理由 / 先读哪里 / 读后收获”

#### 6.3 模块三：会话联动（LLM 辅助导航）

功能介绍：

- 在聊天中回答“这个领域最近有什么新突破/该看哪些文章”时，直接调用学术情报与路径模块

实现草案：

- 在 `generate` 过程中增加“情报查询工具”分支
- 让 LLM 使用结构化结果回答，而非重新从零总结
- 回答中附行动入口：
  - 加入文献篮
  - 打开阅读
  - 生成该方向候选路径

### 7. 数据与接口草案（建议）

#### 7.1 数据模型（最小版本）

建议新增（可先 SQLite，后续再演进）：

- `venues`（会议/期刊基础信息）
- `venue_editions`（某届会议 / 某卷期 / 某时间窗口）
- `venue_papers`（题录、摘要、链接、领域标签）
- `venue_topics`（主题聚类结果）
- `venue_breakthrough_candidates`（突破候选 + 证据 + 置信度）
- `user_profiles`（领域、目标、偏好、预算）
- `learning_path_sessions`（一次候选路径生成）
- `learning_path_items`（路径条目，可排序、可编辑）

与现有模块联动：

- 与 `paper connected`：二级库允许“无全文仅题录”先入库，可直接作为底座
- 与 `文献篮`：推荐论文一键加入
- 与 `Project`：按课题维度保存路径
- 与 Markdown 阅读：路径条目可直接打开阅读面板

#### 7.2 API 草案（示例）

- `GET /api/venues`：可选 venue 列表与筛选条件
- `POST /api/venues/digest/generate`：生成某领域/venue 的情报摘要（异步任务）
- `GET /api/venues/digest/{id}`：获取摘要结果
- `POST /api/learning-paths/generate`：生成用户主导学习路径候选
- `PATCH /api/learning-paths/{id}`：用户编辑路径（排序/替换/标记）
- `POST /api/learning-paths/{id}/explain`：让 LLM 解释某条推荐与用户目标关系

### 8. 实施建议（按风险递增，不写死时间）

#### 阶段一：通用 Digest（先不做个性化）

- 支持一个会议场景 + 一个期刊场景
- 能做领域过滤、主题聚类、代表论文与突破候选展示
- 输出证据化摘要（LLM 解释 + 结构化来源）

#### 阶段二：用户主导路径（显式偏好版）

- 加用户设置（领域/目标/预算/偏好）
- 生成多个候选路径版本
- 支持用户手动调整与重新生成

#### 阶段三：与现有产品深度联动

- 文献篮 / Project / 会话 / Markdown 阅读联动
- 从“推荐”到“阅读”到“追问”的闭环
- 逐步接入 `paper connected` 的二级/三级扩展能力

### 9. 成功标准（用户体验视角）

判断这个功能是否“做对了”，建议看这些：

- 用户能否快速知道“最近哪些方向值得看”
- 用户能否理解“为什么推荐给我”（不是黑盒推荐）
- 用户是否可以轻松改写/重排路径（主导感强）
- 用户是否愿意把推荐论文加入文献篮 / Project
- 推荐是否能自然导向阅读与后续提问（形成闭环）

最终原则（再次强调）：

- LLM 是“增强型研究助手”，不是替用户制定死计划的“学习管理器”。
