## 0. 先给结论：哪些最值得做

### 必做（高价值，直接影响用户）

1. 引用定位正确性与可验证性（尤其图/公式/定理定位）
2. 文献篮状态一致性（切会话、切页面、刷新后不丢、不串、不降级）
3. 会话与文献管理切换稳定性（避免卡死、避免状态覆盖）
4. 主链路回归保障（`upload -> convert -> reindex -> ask`）

### 应做（产品体验拉升）

1. 检索再读（命中原因、阅读建议、section/subsection/snippet）
2. 会话内 Markdown 阅读面板 + 锚点跳转 + 高亮 + 划词追问
3. 文献篮/引用详情交互改造（减少遮挡、信息更可读）

### 暂缓（先不投入大量资源）

1. 完整 PDF 阅读器
2. Streamlit 旧入口大规模重构（`app.py/theme_legacy/runtime_ui`）
3. 跨领域 Venue Intelligence 与大规模“学习路径”系统化能力

## 1. 当前阶段目标（未来 2-4 周）

目标：先把“能稳定读懂并追溯文献”的主链路打牢，再扩功能。

### Phase P0：稳定性与正确性封口

工作项：

1. 修完并锁定转换链路中图/公式映射错误（含 NatPhoton 类 case）
2. 文献篮状态机收敛：本地缓存、会话切换、页面跳转、刷新恢复行为统一
3. 切页面/切会话卡顿治理：减少阻塞式切换、加并发防抖和过期请求保护
4. 新增回归用例：
   - 文献篮跨会话切换不丢失
   - 切换 Chat/Library 往返不清空
   - 连续切换不出现 UI 假死
   - 引用详情字段不降级（刷新后仍保持结构化信息）

验收标准：

1. 至少 3 轮手工回归，问题不可复现
2. 单测/集成测试覆盖关键状态切换路径
3. GitHub CI 全绿

---

## 2. 第二阶段（未来 4-8 周）

### Phase P1：检索再读 MVP

工作项：

1. 引用详情输出结构标准化：`section/subsection/snippet/why/recommend`
2. snippet 默认只显示最相关一句，可展开上下文
3. 引用卡支持“为什么命中”和“建议先读哪里”
4. 页码信息保留为辅助，不做主入口按钮

验收标准：

1. 每条高质量命中都有可解释原因
2. 用户可直接进入建议阅读位置
3. “命中但不知为何”的反馈显著减少

### Phase P2：会话内阅读闭环 MVP

工作项：

1. Chat 页右侧 Markdown 阅读面板（不做完整 PDF 阅读器）
2. 从引用一键跳到锚点位置并高亮
3. 划词追问（ASK）把选中文本 + 文献上下文注入问答
4. 阅读器内“返回会话继续问”链路顺畅

验收标准：

1. 会话 -> 引用 -> 阅读 -> 划词追问闭环可用
2. 关键跳转成功率高（锚点可达、定位正确）

---

## 3. 第三阶段（8 周后）

### Phase P3：文献篮与引用 UI 重构

工作项：

1. 优先保留悬浮框方案，但优化显示/消失与遮挡策略
2. 如果遮挡仍明显，再升级为可调宽右侧抽屉
3. 完善“加入文献篮”反馈（去重、定位、高亮、来源校验）

验收标准：

1. 展开文献篮不明显遮挡主要对话内容
2. 文献信息与 source 实体一致，可追溯

### Phase P4：Paper Connected（分层推进）

工作项：

1. 一级文献规范化（实体 ID、状态、去重）
2. 二级 references 入库（允许仅题录）
3. 三级引用/被引关系扩展（预算可控）

验收标准：

1. 一级文献可查看结构化 references
2. 二级对象可独立浏览
3. 至少一个方向关系可视化可用

## 5. 执行规则（确保推进不断档）

每个阶段都按同一规则执行：

1. 先写最小设计说明（输入/输出/状态/失败路径）
2. 先补测试再改逻辑，至少覆盖一个失败场景
3. 每个 PR 控制在单一主题，避免“功能+重构+样式”混杂
4. 必须有可验证验收项，不用“主观感觉更好了”作为完成标准
