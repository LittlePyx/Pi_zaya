## 1. 需求整理（按主题归类）

### A. 检索再读（高优先级）

- 提升参考定位准确性
- 命中参考文献后细读（兼顾效率、精度）
- 参考定位细化到文献的 `section / subsection`
- 显示原文片段（需要讨论是否保留，若保留的话显示最相关一句）
- 给出“这篇文献讲了什么 / 与当前问题相关性是什么 / 应该从哪里读起 / 用户进入该文献阅读可收获到什么与提问相关的内容”的摘要与导航建议，若太长点击展开
- `page` 按钮（页码入口）是否保留？（倾向于删去）

### B. 文献阅读与阅读指引（高优先级）

- 增加文献阅读指引能力（从会话入口进入、从参考定位进入）
- 倾向基于 Markdown 阅读，而不是先做完整 PDF 阅读器
- 会话中若提到相应的文献中的地方 / 定位就在附近提供一个小入口，点击后右边可打开 Markdown（以HTML渲染）HTML DOM+锚点 ，支持跳转 + 高亮
- 在阅读页面（以HTML显示的Markdown）支持“划词提问”（监听 `mouseup`，可直接点击出现的小悬浮框“ASK”把选中文本送入提问流程）
- 阅读时可解释文中具体句子/段落含义（偏“边读边问”）

### C. 文献篮 / 引用 UI（高优先级）

- 核对悬浮框里的 source 信息是否与文献实体准确对应
- 二级文献入库时，引用标题与实际文献/Markdown 标题的映射对齐
- 右侧悬浮框会遮挡对话，需要优化交互形态
- 候选方案：
  - 改为可调节宽度右侧栏（类似左侧边栏）
  - 改为单独页面
  - 保留悬浮框但优化显示/消失逻辑（暂时倾向于这个）

### D. Paper Connected（文献关系网络，分阶段做）

- 一级：用户主动上传文献（当前已有基础）
- 二级：把一级文献的 references 做入库（可基于 Crossref）
- 二级库最好做成独立模块，可展开/关闭查看
- 三级：在二级基础上扩展“引用 + 被引”关系，逐步形成 `paper connected` 网站/图谱
- 可在会话中选择打开（？）

### E. 会话能力增强（高优先级）

- 输出与回答相关的文献图片，并结合图注/正文解释
- source 粒度更细（公理、公式这种都需要能显示source）
- 内容质量控制（减少跑偏、提高稳定性）
- 代码/伪代码渲染美化与高亮
- 会话中支持选字触发悬浮框，继续 ASK（类似 GPT 阅读交互）
- Timeline：鼠标悬停显示当时提问内容
- 用户需求：如果用户问到某一篇文章种的某一个定理（公式）该如何解释/证明，需要有定位到的能力。

### F. PDF -> Markdown 质量与工程化（高优先级）

- 用 1980 年前扫描文献验证转换稳定性（判断是否要预处理）
- 用数学/公式密集文献做专项验证
- 评估“翻译后 Markdown”功能是否需要
- 代码过长，需要整理
- 结合阅读场景，把图像抽取能力接入会话输出  

### G. 用户数据 / 项目管理 / 文献管理增强（中高优先级）

- 用户数据库（用户级数据持久化）
- 历史会话管理
- Project 概念（按项目组织文献与对话）
- 文献 tag / 分类
- 文献管理页面 UI + 交互优化（含进度条、转换状态显示）
- 文献详细数据获取（题录/元数据增强）
- 面向领域调研/知识库 LLM 的工作流能力

## 2. 总体推进策略（建议）

先做这三件，再扩功能：

1. 打通“检索再读 + 文献阅读（Markdown）”主链路
2. 再扩展 `paper connected` 和高级会话能力

原因：

- 你的核心价值已经不是“能问答”，而是“能快速、可信地读懂文献”
- `检索再读`、`文献阅读引导`、`source 细粒度` 直接提升用户体验和学术可信度
- `paper connected` 很有潜力，但对数据模型和引用质量依赖大，适合在引用定位稳定后推进

## Phase 2：文献阅读（Markdown）与划词问答 MVP

目标：形成“会话 -> 引用 -> 阅读 -> 划词追问”的闭环。

范围控制（重要）：

- 暂不做完整 PDF 阅读器（缩放/双击/复杂交互先不做）
- 优先做会话右侧 Markdown 阅读面板（更贴近你当前使用习惯）
- 独立 ReaderPage 放到后续（不是本阶段硬要求）

工作项：

- 在会话命中附近提供“小入口”（提到/命中文献时可点击）
- 点击后在右侧打开 Markdown 阅读面板（HTML 渲染 + DOM 锚点）
- 从引用点击进入对应文献、对应位置（anchor/段落）
- 高亮当前命中片段
- 阅读侧边栏（目录 / 命中点列表 / 引用跳转历史，可简化版先上）
- 划词问答交互（监听 `mouseup`）
  - 选中文字后出现小悬浮框 / 按钮（ASK）
  - 一键发起“解释这段 / 结合上下文说明 / 与当前问题关系”
- 把阅读上下文（文献名、section、选中文本）注入提问 prompt

前端落点（建议）：

- 新增 `web/src/components/reader/*`（先做右侧面板）
- 新增 `web/src/components/reader/ReaderPanel.tsx`（建议）
- `web/src/pages/ChatPage.tsx`（会话内阅读入口与右侧联动）
- `web/src/App.tsx`（路由）
- `web/src/stores/chatStore.ts`（增加“从阅读器提问”动作）

后端落点（建议）：

- `api/routers/generate.py`（支持阅读器上下文字段）
- `api/routers/references.py`（提供 reader 跳转定位信息）

验收标准：

- 会话中可通过“小入口”在右侧打开文献阅读面板
- 能从会话引用直接跳到 Markdown 对应位置（DOM 锚点）
- 命中片段高亮可见
- 划词后可一键发起追问并收到回答

## Phase 5：Paper Connected（一级 -> 二级 -> 三级）分层推进

目标：从“单文献检索问答”扩展到“文献关系导航”。

### 5.1 一级（已有基础，先补规范）

- 用户上传文献入库流程规范化（状态、去重、元数据）
- 文献实体统一 ID（为二级/三级扩展做准备）

### 5.2 二级（references 入库）

- 解析一级文献 references
- Crossref / 本地规则匹配题录元数据
- 生成二级文献实体（允许“无全文仅题录”先入库，已确认）
- 二级文献展示为独立模块（可展开/关闭）

### 5.3 三级（引用 + 被引拓展）

- 基于二级文献继续获取引用/被引关系（预算可控）
- 构建轻量文献关系图视图（先列表/树，后图谱）
- 形成 `paper connected` 页面雏形
- 在会话中提供可选入口打开 `paper connected`（实验入口/开关）

建议先做的数据模型（优先）：

- `paper`
- `reference_edge`
- `citation_edge`
- `source_confidence`
- `import_status`

验收标准：

- 一级文献可看到结构化 references
- 二级文献（含“仅题录无全文”）可作为独立对象浏览
- 至少支持一个方向（引用或被引）的关系扩展显示
